## 关键实现要点

### JWT 认证（单体版，Plan 05 前 gateway 暂不独立）

```go
// internal/user/auth/jwt.go
// 签发 access + refresh token
// 中间件：解析 Bearer token → ctx 写入 userID + roles
```

- access TTL / refresh TTL 与 NestJS 配置一致
- GitHub OAuth：`golang.org/x/oauth2`
- 验证码：`mojocn/base64Captcha`，Redis 存 session

### 模块间 interface（为 Plan 05 gRPC 做准备）

```go
// internal/user/port.go
type UserService interface {
    GetUser(ctx context.Context, id uint64) (*UserDTO, error)
    GetUserBatch(ctx context.Context, ids []uint64) ([]*UserDTO, error)
}
```

Plan 02-04 内其他模块（如 article 取作者）依赖此 interface 的**本地实现**。

### RBAC

- 角色-权限-菜单树 CRUD
- 接口级权限校验中间件（对照 NestJS `@RequirePermission`）
- 数据权限（data-scope）若 NestJS 有实现，本阶段同步迁移

## 生产切换提示

本计划完成后，可按总方案 8.5 **绞杀者模式**将用户相关路径 Nginx 灰度切到 Go 单体：

```
/api/v1/auth  → Go
/api/v1/user  → Go
/api/v1/captcha → Go
/api/v1/admin → Go
```

其余路径仍走 NestJS，逐步扩大。

## 风险与注意点

| 风险 | 对策 |
|------|------|
| 旧密码哈希不兼容 | 对照 NestJS crypto 逻辑，写集成测试用现网样本 |
| RBAC 权限码不一致 | 从 NestJS 数据库导出 privilege 表对照 |
| OAuth redirect URI | 开发/生产 config 分环境配置 |
