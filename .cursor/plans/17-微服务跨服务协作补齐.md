| 项 | 内容 |

## 「跨服务协作」是什么意思？（不是浏览器 CORS）

拆成 gateway / user / blog / rpg 四个进程后：

- **表归属**：敏感词表在 user-service，评论/文章在 blog-service，RPG 在 rpg-service（见总方案 3.3）。
- **禁止**：blog-service 的 Ent **不能** JOIN `x_sensitive_word` 或 `x_user` 表。
- **必须**：blog 发评论时要 **gRPC 调 user** 做敏感词过滤；admin 文章列表要 **gRPC 调 user** 解析数据权限与部门名。

Plan 11 为快速跑通微服务，部分能力用了 **Noop / Permissive / 空列表 stub**。本计划把这些 stub 换成与 Nest 单体等价的行为。

```
┌─────────────┐     gRPC      ┌─────────────┐
│ blog-service│ ────────────► │ user-service│  敏感词过滤、部门、数据权限、锁定用户列表
└─────────────┘               └─────────────┘
       │ gRPC
       ▼
┌─────────────┐
│ rpg-service │  hit-records（读 user 域 hit 表）
└─────────────┘
```

## 模块范围

### A. 敏感词过滤（user → blog）

| 现状 | 目标 |
|------|------|
| `blog/internal/contentfilter/noop.go` 透传，评论/留言/回复**不过滤** | user gRPC `EvaluateContent` + `CreateHitRecord` |
| blog `providers.go` 注入 `NoopFilter` | 注入 `GRPCFilter` 调 user-service |

proto 建议（`proto/user/v1/sensitive.proto` 或扩展现有 `user.proto`）：

- `EvaluateContent(text) → { content, hitWords, hpPenalty, needReview, rejected }`
- `CreateHitRecord(params) → ok`

### B. 数据权限（user → blog admin 文章）

| 现状 | 目标 |
|------|------|
| `PermissiveArticleAdminPort` 不限制 dept | gRPC `ResolveAccessibleDeptIDs(uid, resourceType)` |
| `AssertArticleDeptAccess` 恒 true | 对齐 Nest `DataScopeService.canAccessDept` |

user-service 已有 `AdminRepo.ResolveAccessibleDeptIDs`；暴露 gRPC 即可。blog `ArticleService.List/Edit/Delete` 已调用 port，换实现即可。

### C. C 端作者过滤（user → blog 列表）

| 现状 | 目标 |
|------|------|
| `ListActiveUserIDs` 返回 nil = 不过滤 | gRPC 返回 `status=ACTIVE` 且未删除的用户 ID 列表 |

对齐 Nest：C 端文章列表不展示锁定/禁用作者的内容。

### D. 部门名（user → blog 详情/列表）

| 现状 | 目标 |
|------|------|
| `FindDeptByID` 报错 | gRPC `GetDept(id) → { deptName }` |

修复文章列表/详情 `deptName` 为空。

### E. 敏感词命中记录（user → rpg C 端）

| 现状 | 目标 |
|------|------|
| `rpg/internal/app/adapters.go` `hitRecordsStub` 空列表 | gRPC `ListSensitiveWordHits(uid, page, pageSize)` |

表在 user 域；rpg-service **禁止**本地 Ent 查 hit 表（Plan 11 已知限制）。

### F. 审核后同步实体状态（user → blog，与 Plan 13 衔接）

| 现状 | 目标 |
|------|------|
| `syncSourceStatus` 仅打日志 | blog gRPC `UpdateModerationStatus(sourceType, sourceId, approved\|rejected)` |

Plan 13 验收依赖本项；若 13 先做，须至少完成 F。

## 任务清单

- [ ] 扩展 `proto/user/v1`，buf generate
- [ ] user-service 实现 gRPC server 方法（复用现有 sensitive/admin repo）
- [ ] blog-service：`GRPCFilter` 替换 Noop；`GRPCArticleUserPort` / `GRPCArticleAdminPort` 替换 Permissive
- [ ] rpg-service：`hit-records` 调 user gRPC
- [ ] blog gRPC：`UpdateModerationStatus`（comment/msgboard/reply 按 sourceType 更新 status）
- [ ] 单元测试：Noop 移除后 comment create 命中敏感词 → pending
- [ ] 集成测试：非超管 admin 文章列表 dept 过滤
- [ ] 更新 `docs/api-routes.md` gRPC 表

## 验收标准

- [ ] blog-service 评论含敏感词 → `status=pending` 或拒绝（与 Nest 一致）
- [ ] admin 文章列表：DEPT 数据权限角色只能看到本机构文章
- [ ] C 端列表：锁定用户文章不出现
- [ ] 文章列表 `deptName` 有值（用户有关联 dept 时）
- [ ] `GET /rpg/hit-records` 返回当前用户真实命中分页
- [ ] hit approve 后 comment status 变更（与 Plan 13 联调）
- [ ] grep 无 `NewNoopFilter()` / `PermissiveArticleAdminPort` 生产路径

## 风险与注意点

| 风险 | 对策 |
|------|------|
| gRPC 增加评论延迟 | 敏感词结果短 TTL 本地 cache（可选）；MVP 先直调 |
| user gRPC 不可用 | blog 降级策略写进交付文档（fail closed vs open） |
