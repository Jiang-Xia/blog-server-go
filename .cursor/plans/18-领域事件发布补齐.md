# Plan 18：领域事件发布补齐

## 元信息

| 项 | 内容 |
|----|------|
| **目标** | blog-service 在业务写路径向 Redis Stream `blog:events` 发布领域事件，与 rpg-service 已有消费者闭环 |
| **前置依赖** | [17-微服务跨服务协作补齐.md](./17-微服务跨服务协作补齐.md) 验收通过（敏感词命中需带 `hpPenalty` 发事件） |
| **周期** | ~1–1.5 周 |
| **架构形态** | blog-service 发布 + rpg-service 消费（现有 `rpg/internal/rpg/event/consumer.go`） |
| **里程碑** | M6 Nest 差异补齐 |
| **Nest 对照** | `modules/core/events/blog.events.ts` + 各 feature `BlogEventPublisher.publish` |

## 背景

Plan 08 搭建了 Stream 框架；Plan 09 在 **rpg-service** 注册了完整消费者（发文经验、评论任务、敏感词扣 HP 等）。

但 **blog/monolith 的业务 service 几乎不调用 `Publisher.Publish`**（仅 dev 调试接口与少量 rpg 侧 tip 活动）。导致：

- 签到以外的 **任务/成就进度** 不更新
- **敏感词扣 HP / 禁言** 不触发（Nest 在 comment 创建时发 `SENSITIVE_WORD_HIT`）
- **定时发布**（Plan 12）发布后 RPG 无经验

本计划补齐 **发布端**，消费端增量在 Plan 13 完善 `PunishmentService` / `ArticleLevelService`。

## 事件清单（对齐 Nest `BlogEvents`）

| 事件 type | 发布时机 | payload 要点 | 主要消费者 |
|-----------|----------|--------------|------------|
| `blog.article.published` | 状态→publish、定时发布 job | `uid`, `articleId` | RPG 经验/任务/成就；文章等级（Plan 13） |
| `blog.article.updated` | 已发布文章编辑 | `uid`, `articleId` | 索引/RAG（Plan 15） |
| `blog.article.unpublished` | 下架 | 同上 | RAG |
| `blog.article.deleted` | 软删 | 同上 | RAG |
| `blog.article.viewed` | 阅读 +1 | `uid?`, `articleId` | RPG 浏览任务 |
| `blog.comment.created` | 评论入库 | `uid`, `articleId`, `commentId` | RPG 经验/任务 |
| `blog.reply.created` | 回复入库 | `uid`, … | 同上 |
| `blog.msgboard.created` | 留言入库 | `uid?` | 同上 |
| `blog.like.created` | 点赞 | `uid`, `articleId` | RPG |
| `blog.collect.created` | 收藏 | `uid`, `articleId` | RPG |
| `blog.sensitive-word.hit` | 命中且需扣 HP | `uid`, `hpPenalty` | RPG `PunishmentService` |

> 发布失败 **只打日志、不阻断主流程**（与 Nest `BlogEventPublisher` 一致）。

## 模块范围

### blog-service 改造

| 包 | 改动 |
|----|------|
| `internal/blog/service/article_service.go` | create/edit/delete/disabled/views 发事件 |
| `internal/blog/service/comment_service.go` | create 发 comment + sensitive-word.hit |
| `internal/blog/service/reply_service.go` | 同上 |
| `internal/blog/service/msgboard_service.go` | 同上 |
| `internal/blog/service/like_service.go` / `collect_service.go` | 发互动事件 |
| `internal/event/publisher.go` | 已有，wire 注入上述 service |
| Plan 12 `scheduled_publish` job | 发布成功后发 `article.published` |

### rpg-service 增量（本计划或 Plan 13）

| 项 | 说明 |
|----|------|
| `onSensitiveWordHit` | 对齐 Nest：护盾、累计禁言、归零禁言、WS 通知（Plan 13 可拆） |
| `onArticlePublished` | 接入 `ArticleLevelService`，写入 `article.articleLevel` / `articleExp` |
| `article.published` blog 侧 consumer | 可选：刷新统计缓存（Plan 08 遗留 debug 消费者） |

- [x] 发布文章后 RPG 用户经验 +20（或配置值），任务 `article` 进度 +1
- [x] 发评论后经验 +5，`comment` 任务进度 +1

- [x] 点赞/收藏触发对应 RPG 任务（若 Nest 有）
- [x] dev 以外路径不依赖 `/dev/event-publish`
- [x] monolith 副本若仍保留，同步改或标记 deprecated

### 可脚本化验收

```bash
make dev-all
# 登录用户发评论
curl -sf -X POST http://127.0.0.1:8000/api/v1/comment/create \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"articleId":1,"content":"测试评论"}'

# 查 RPG 经验是否变化
curl -sf http://127.0.0.1:8000/api/v1/rpg/status -H "Authorization: Bearer $TOKEN"
```

## 本计划不做

- Stream 新中间件（仍用 Redis Stream）
- 全量 Nest 事件名以外的自定义事件
- RAG 索引 consumer — Plan 15

## 风险与注意点

| 风险 | 对策 |
|------|------|
| 重复 publish（编辑多次） | 与 Nest 相同语义即可；RPG 侧幂等按 event+id 可选 |
| views 高频 | 与 Nest 一致先每次发；后续可加 Redis 去重 |
